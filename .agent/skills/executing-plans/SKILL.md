# Executing Plans

## Overview

This skill defines the strict `AGENTS.md` BUILD and QA states. You will load the approved plan, execute the requested tasks in a sandbox, run verifications, and present the diff for Human Approval. 

**Core Principle:** No fake/simulated data. No applying changes to `main` without approval. Stick to the budget. Context must remain focused only on task-relevant files.

**Context:** Triggered when the user says "approved" or "looks good" in response to a plan generated by `writing-plans`.

## Context Management

When entering the BUILD state, you must drop any files loaded during the PLAN state that are not immediately relevant to the specific build phase you are starting.
- **Reference (on-demand):** Do not load the entire codebase. Grep for patterns only.
- **Task (current task):** Load only files being modified, direct dependencies, and tests.

## The BUILD Process

**Substate:** Set to `CODING`

For each component outlined in the approved Plan:
1. Work in a branch or temporary clone (never main).
2. Create/modify files implementing minimal changes achieving the objective.
3. Follow patterns established in `memory-bank/projectRules.md` or existing code.
4. Add tests alongside your implementation.
5. Generate a unified diff of your changes.
6. **DO NOT APPLY CHANGES TO MAIN.**

### Context Focus Loop
If the task has multiple independent steps, rotate your context. After finishing Step 1, close the files unrelated to Step 2 before starting Step 2.

### Stall Detection (Red Flag)
If you generate two consecutive identical diffs (i.e. you try the same exact compile fix twice and fail both times), you must throw a `STALL DETECTED` warning and stop executing.

```markdown
## STALL DETECTED

⚠️  Two identical diffs - unable to progress

**Diagnosis**:
- Cause: [specific technical reason]
- Attempted: [what was tried]

**Request**: Provide direction or choose recommendation
```

## The QA Process

Once the BUILD state is complete and the unified diff is generated:

**Substate:** Set to `RUNNING`

You must execute the following checks automatically:
1. Run the test suite relevant to your changes.
2. Run linters and code quality checks.
3. Verify the build.

If tests fail, fallback to BUILD and synthesize a minimal patch. 
If tests fail 3 times in a row, trigger the `STALL DETECTED` warning and request user input.

## Presenting for APPROVAL

When the BUILD is complete and QA has passed, you must halt execution and present the user with the APPROVAL gate.

Output this exact format:

```markdown
## Ready for Approval

Code changes complete. Ready for review.

**Files modified**:
- `path/file1.ext` (+50, -10 lines)
- `tests/test.ext` (+200, -0 lines)

**Test Results**:
✅ tests passing | ✅ Linter clean | ✅ Build successful

**Review Gates**:
- ✅ Tests pass
- ✅ Security reviewed (no sensitive data, validated inputs)
- ✅ Documentation plan ready

**Next Steps After Approval**:
1. Apply changes to sandbox branch

---

**Please review. Reply with**:
- "approved" | "looks good" | "document it" -> Proceed to APPLY
- "change X" | "fix Y" -> Return to BUILD with changes
- "revert" -> Discard all changes
```

## DO NOT PROCEED

You must explicitly end your response and wait for the user to type "approved", "looks good", or similar. If the user does not explicitly approve, you must not apply the changes.
